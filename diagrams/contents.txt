Especificación para Diagrama TikZ: Flujo y Vehículos de Comunicación de la PlataformaInstrucción para el LLM / Copilot:Genera un diagrama en LaTeX utilizando TikZ (sugiero las librerías positioning, arrows.meta, shapes.geometric y opcionalmente pgf-umlsd si decides usar un formato estricto de secuencia). El diagrama debe representar un híbrido entre un diagrama de secuencia y un diagrama de flujo de datos, organizado en columnas (carriles/lifelines) de izquierda a derecha. Cada flecha debe tener una etiqueta explícita que denote el "Vehículo" de comunicación (ej. REST, Celery, ORM).1. Definición de Nodos Principales (Columnas / Carriles)Organiza los siguientes nodos horizontalmente de izquierda a derecha. Representa las bases de datos con forma de cilindro y los servicios con rectángulos.A - Frontend (SPA UI): Representa el TaskService y las UI Gateways.B - GitHub: Servicio externo (Almacenamiento y GitEngine).C - Backend Gateway: Representa los API Controllers y el Backend Service Core.D - PostgreSQL: Base de datos relacional.E - Redis (Broker & Result Backend): Sistema de colas en memoria.F - Lean Celery Worker: El microservicio de validación.2. Flujo 1: Autenticación y Sincronización de Perfil (Flujo Síncrono Externo)Este flujo describe cómo un usuario ingresa y cómo el sistema se comunica con el exterior.Paso 1.1: Flecha sólida de Frontend a Backend Gateway.Acción: Solicitar inicio de sesión o validación de token.Vehículo: REST / HTTP JSON.Paso 1.2: Flecha sólida de Backend Gateway a GitHub.Acción: OAuth Callback / Obtener perfil de usuario.Vehículo: REST + OAuth.Paso 1.3: Flecha de retorno (punteada) de GitHub a Backend Gateway.Acción: Retorna datos del perfil y tokens de acceso.Vehículo: HTTP Response (JSON).Paso 1.4: Flecha sólida de Backend Gateway a PostgreSQL.Acción: Persistencia de usuario (Upsert en la tabla de usuarios).Vehículo: SQL ORM.Paso 1.5: Flecha de retorno (punteada) de Backend Gateway a Frontend.Acción: Entrega de credenciales de sesión al cliente.Vehículo: HTTP Response con cabecera authHeaders(Bearer JWT).3. Flujo 2: Modificación del Grafo Matemático (Flujo Transaccional Interno)Este flujo representa operaciones de dominio como crear proyectos, dividir nodos (splitNode) o hacer merge de Pull Requests.Paso 2.1: Flecha sólida de Frontend a Backend Gateway.Acción: Invocación de casos de uso (Directo), ej. guardar un nuevo lema.Vehículo: REST / HTTP JSON autenticado con JWT.Paso 2.2: Flecha sólida de Backend Gateway a PostgreSQL.Acción: Actualizar el estado del nodo y transiciones del grafo.Vehículo: SQL ORM.Paso 2.3: Flecha sólida bidireccional entre Backend Gateway y GitHub.Acción: Operaciones de repositorio, commit y persistencia en rama.Vehículo: REST API de GitHub.Paso 2.4: Flecha de retorno de Backend Gateway a Frontend.Acción: Confirmación de éxito (HTTP 200 OK).4. Flujo 3: Verificación de Pruebas (Flujo Asíncrono Desacoplado)Este es el flujo crítico que demuestra el patrón de microservicios, donde el vehículo cambia drásticamente para proteger el rendimiento del servidor.Paso 3.1: Flecha sólida de Frontend a Backend Gateway.Acción: Solicitar verifyNode.Vehículo: REST / HTTP JSON.Paso 3.2: Flecha sólida de Backend Gateway (específicamente del Compiler Client Gateway ) hacia Redis.Acción: Enviar tarea tasks.verify_snippet o verify_project_files.Vehículo: Protocolo Celery a través de la cola en memoria lean_queue.Paso 3.3: Flecha de retorno inmediata de Backend Gateway a Frontend.Acción: Confirmación de recepción.Vehículo: HTTP 202 Accepted (Retorna Job ID).Paso 3.4: Flecha sólida desde Redis hacia Lean Celery Worker.Acción: Consumo de la tarea (Routing tasks).Vehículo: Suscripción asíncrona de Celery Worker.Paso 3.5: Flecha circular sobre Lean Celery Worker (Loop local).Acción: verify_lean_proof y parse_lean_messages.Vehículo: Invocación Directa (Ejecución de procesos a nivel de sistema operativo y adaptadores locales).Paso 3.6: Flecha de retorno de Lean Celery Worker a Redis.Acción: Almacenamiento del resultado formatado (to_compiler_snippet_response).Vehículo: Celery Result Backend (Escritura en base de datos en memoria Redis).Paso 3.7: Flecha sólida de Backend Gateway a Redis.Acción: El servidor lee el resultado final alojado en Redis.Vehículo: Consulta directa a Redis.Paso 3.8: Flecha punteada de Backend Gateway a Frontend.Acción: Notificación de tarea completada al cliente.Vehículo: Eventos enviados por el servidor (SSE / WebSockets, derivado del TaskStreamService y SSE EventSource stream ) o Webhook JSON.
